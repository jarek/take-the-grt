Developers/Todo

From Take the GRT

Jump to: navigation, search

Items lower are not necessarily less important, though they may come later in the process.

  1. Translate the Google transit schedule format into an SQL database. Perhaps add a field about valid since/valid until data so that we can roll over schedules more elegantly. (How neat would "The GRT is going to be changing their schedules on December 28th, 2008. Would you like to set up a reminder to update your data / update it automatically once it becomes available?" be? Super neat.) The SQL database is mostly done. The valid since/until data is part of the GTFS spec.
    While we're at it, write code to take data (or a subset - eg, only one or a couple of routes) and write it in the Google format (this should be all but trivial). We're looking at the google text format for the system/route data to be downloaded and cached on the devices, and the XML to be accessible over a web service for more bandwidth-rich mobile apps, always-on mobile apps, and web mashups.
  2. Populate the database. If we have more hands than brains (to speak bluntly), do it brute force out from PDFs. If everyone does the route they take + one more, it'll take 2-3 hours per person and 20-ish people. Definitely doable. If possible, also create/start creating a programmatic way of doing this, either through scraping the GRT schedule site (some POST hackery required - looks like at least the session cookie), or some mad PDF hack. Create a quick way of checking the GRT website to see if they have posted any news regarding schedule changes occured or occuring, and also a way to do this manually for the first couple of times. In the future: a way to hook in a feed of some sort when GRT (hopefully) tells us about the changes.
  3. (In some degree of parallel with 2.) Start working on the applications, at least one or two to start us off. Saying "look at us, we have this data" is a lot less impressive than saying "look at us, we have these sick apps"
  4. Data access: http://serve.takethegrt.ca/ or something similar. Whole system GTFS zip, K-W zip, Cambridge zip, an interface to only get the (fully correct) zip for one or more routes so that we have to download and store (cache) less. Second prority: a sensible format for the XML and code to serve the information in this format. Can be a pet project for an XML wonk (and I use the term lovingly).
  5. Graphic design for our site/logo/overall process / pet or "featured" applications. We want this to look presentable. Good for a UI wonk. How would it make sense to present this stuff? user:Qviri of course has ideas for the Blackberry app, but I'm sure they're far from the end-all-be-all.
  6. Marketing/spread the word. People are riding the GRT. People have cell phones. The two go together. Students are riding the GRT (we want them to! That's the pass is for). They have cell phones, laptops, lab computers.
    Make sure we're NOT going against the GRT. We are merely supplenting them. They run the system of awesomeness. After a certain tipping point, we could probably win a PR battle, but we don't want to have that battle in the first place.
  7. After we get serious enough (two-three popular apps, thousands of users, newspaper reports), talk to GRT about letting us on the inside of the process/legimizing us more or less/telling Google to use our data (or their copy of it) for the Maps system.
  8. Part of 7, really. Ask for access to data they don't post publicly: eg, feeds from GPS devices on ixpress and whereever else they get installed by then. Look into possibilities for getting even more data â”€ eg, how many people on a given bus (run)? How late is a bus running? On average would be good. Instanteously would be better. No idea how this could happen ATM, but a neat idea.
    1. If not this, then generate some of the data ourselves. For instance, live on a bus route? See buses from your window? Put up a webcam, run software that will look at the snapshots and attempt to detect the bus, possibly read the route/destination sign if we're feeling really fancy. Ping our server with the info and we can have real time bus pseudo-tracking as they move along a route. Would need lots of webcams to really replace GPS, but could have use for other purposes - for example, a webcam at the beginning of Keats can tell us how late a given run is, we can then display the info on the schedule for subsequent stops: "15:17 (running 5 min late)". If we have enough consistent data points, we could try to stitch together an alternate, actually realistic schedule. Either way, should be a fun exercise, if not central to our goal.
  9. Backwards compatiblity! Version 1.0 of that Java MIDP app should optimally never break as long as our server is on the internets.
