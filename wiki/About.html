About

From Take the GRT

Jump to: navigation, search

The Grand River Transit system is an important asset to the Waterloo Region. As the region strives to improve its reputation as a friendly, desirable place to live, and especially as it tries to attract high tech employers - and employees - it becomes important to make the GRT technology friendly.

Currently, the GRT.ca website, in addition to information about fares, important announcements, etc, contains copies of bus route schedule pamphlets. These are PDF files on average about 1.5 MB in size. The site also features a trip planner powered by proprietary technology from GIRO. This planner also includes a feature that enables viewing of bus schedules, per route and stop. Both of these require a browser which supports Ajax and submitting forms using POST. The first requirement eliminates the majority of mobile phone browsers from using the site. The second makes it a lot harder for third-party applications to use the schedule data. To date, we don't know of an application other than the GRT/GIRO trip planner that uses this data.

Making the data available more easily would open up a plethora of opportunities for improved rider experience. If the data can be downloaded once and cached locally, mobile applications would work without requiring an active internet connection or accumulating per use charges â”€ a factor especially in Canada where the cellular bandwidth prices are some of the highest in the world. With a lot of devices equipped with GPS, an application could locate the nearest stop automatically and use that information when creating a route in a third-party trip planner (automatically using GPS data is currently not possible with the GRT's trip planner). If a user usually rides only one or two routes and uses only a few stops, a third-party application could allow him to easily check when the next bus is coming without having to navigate through a list of all the routes and stops on the GRT website.

Currently, the only comprehensive map of the GRT system available online is a pair of PDF files, one for Kitchener-Waterloo and one for Cambridge part of the system. These files are very large (well over 10 MB each) and take a long time to download and open on all but the best internet connections and fastest computers.

The GRT has stated they have been holding discussions regarding integrating the data into the Google Transit system such that it can be displayed on the familiar Google Maps interface, but can offer no concrete timelines. They have also stated there are currently no plans to make the route and schedule data available in machine-readable format for use in third-party applications, although it is something they continue to look into.

While we appreciate that such issues move slowly in large organizations such as the GRT, we believe that the issue could definitely be handled faster. Years spent waiting for machine-readable data and route information in Google Maps are years wasted when it comes to building ridership among the technology-savvy residents of the Waterloo Region. We hope to make this happen much faster by leveraging the technology expertise which already exists in the region.

Our basic goals are two-fold:

  • Extract the data from the existing GRT systems and store it in a machine-readable format. The extraction process will hopefully be automated so that it can be easily repeated when the GRT schedules change, but for the first proof-of-concept implementation, manual copying of data out of the PDF schedules is acceptable. The current plan is to store the data in a relational database modeled relatively closely after Google's transit information format, so that it will be trivial to export it for use by applications. The Google format is an open, freely available specification and using it avoids reinventing the wheel.
  • Create applications that then leverage this data. The sky and the programmer's imagination are the limit here. Just a few of the sample applications are:
    • A generic Java MIDP application that would enable the vast majority of cell phones to access and display schedule data
    • A Blackberry app that would be able to use the GPS sensor present in some Blackberry smartphones to locate the nearest spot and immediately display the next bus time. It would conceivably be possible to use the push email capabilities of Blackberries to automatically alert users when the bus schedule changes to avoid passively polling to see if the data in the application is up to date
    • An iPhone app that could leverage the phone's large screen and speedy rendering capabilities to overlay stop data on top of a map - or a satellite photo - of the area
    • Desktop widgets for Windows Vista, Mac OS X, or any other system or application that supports widgets that would display next departures from a user's usual route and stop
    • A web site "mash-up" overlaying a internet mapping website such as Google Maps or Windows Live Maps with stop information and a third-party trip planner
    • A centralized trip planner that would run on our web server and be accessible to any application - and improved and extended by all interested - to avoid duplication of code in separate applications
